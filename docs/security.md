# セキュリティ

このSDKのセキュリティ上の設計と注意点を、実装を根拠に示す。

## マスク鍵

client ロールで送るフレームのマスク鍵は、フレームごとに `sys_getrandom`（getrandom syscall）由来の CSPRNG から取る（RFC6455 §5.3）。
getrandom が失敗したときは、予測可能な鍵でフレームを送らず、送信失敗（戻り値 0）を選ぶ。
この分岐は `src/sdk/conn.c` の `build_frame` にある。

## 入力検証（信頼境界）

server は unmasked のクライアントフレームを拒否する（§5.1）。
RSV 非0、未知 opcode、非最小長エンコード、64bit 長の最上位ビット、許容量を超える制御フレームも拒否する。
これらの判定は形式検証で網羅性を証明済みであり、すべてのフレーム受信は accept か fail に決定的に分類される。

## UTF-8 検証

text フレームと close reason の不正な UTF-8 は fail（close code 1007）にする（§8.1）。
overlong エンコード、サロゲート、範囲外コードポイントを排除する。
バリデータの健全性と完全性は Lean で証明済みである。

## close code

受信した close code は §7.4.1 の許容域で検証し、域外なら 1002 で fail する。
送信時は予約コード 1005/1006/1015 をフレームに載せない（該当する場合は 1000 に丸める）。

## Fail the Connection（§8）

プロトコル違反を検出するとエラーになり、close を経て CLOSED に至る。
この遷移は一方向であり、状態機械の単調性として証明済みである。

## TLS 未対応

このSDKは平文 WebSocket のみを扱う。
wss://（TLS 上の WebSocket）は実装していない。
本番でインターネットに晒すときは、前段で TLS 終端（リバースプロキシなど）を置くことが必須である。

## freestanding の信頼境界

libc に依存せず syscall を直接発行する。
デモサーバは固定長の接続テーブル（`MAX_CONN` = 64）を持ち、それを超える接続は受理しない。
これは DoS に対する単純な上限であって、本格的なレート制限やタイムアウトは実装していない。

## 脆弱性の報告

脆弱性を見つけたときは、リポジトリの issue で報告してほしい。
